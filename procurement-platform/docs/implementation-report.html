1. Abstract

ProcureX is a modular procurement platform designed to support the full lifecycle of public and private procurement operations. The system includes public marketplace discovery, account registration, company verification, tender planning, tender publication, supplier bidding, bid evaluation, award recommendation, contract negotiation, post-award tracking, communication, records management, and platform compliance administration.

The implementation uses a production monorepo named procurement-platform. The frontend is implemented with React, Vite, Material UI, Redux Toolkit, React Router, i18next, and supporting shared UI utilities. The backend is implemented with TypeScript, Express, Prisma, modular API boundaries, and row-level security foundations. The original procurex-ui prototype is retained as the visual and workflow reference, while the production React client owns the operational implementation.

Documentation note: This report is a documentation deliverable only. It does not introduce public API changes, database schema changes, or application behavior changes.

2. Introduction

Procurement systems require strong traceability, controlled access, structured workflows, and clear collaboration between buyers, suppliers, evaluators, approvers, and administrators. ProcureX addresses these needs by organizing procurement activity around companies and workflow assignments rather than treating buyer and supplier as simple login roles.

The system separates presentation reference from production execution. The prototype in ../procurex-ui captures the intended user experience, while procurement-platform/client implements the production interface. This separation allows the project to preserve the design intent while progressively adding authentication, routing, state management, localization, API integration, tests, and database enforcement.

3. Objectives

The implementation objectives were defined around system completeness, maintainability, and future production readiness.

  1. Implement a production workspace that reflects the ProcureX reference user interface.
  2. Provide a modular backend where each procurement domain has a clear API boundary.
  3. Model identity, organizations, buyer and supplier capabilities, tenders, bids, evaluation, awards, contracts, records, and compliance data.
  4. Separate account type from procurement capability so a company can act as buyer, supplier, or both.
  5. Prepare row-level security foundations for organization-aware data access.
  6. Support screenshot-based UI parity validation between the reference prototype and the production React client.
  7. Provide tests and scripts that support local development, validation, database setup, and future enhancement.

4. Methodology

The development approach used a reference-driven migration methodology. The original static ProcureX interface remained the product and visual source of truth. Production implementation then translated that reference into React pages, guarded routes, domain modules, database models, and repeatable tests.

4.1 Reference UI Method

UI parity is maintained by capturing screenshots from both the static prototype and the production React target. This makes design drift visible and gives the team concrete evidence for report, review, and validation work.

4.2 Modular Development Method

Backend modules follow a consistent structure: 

routes.ts -> controller.ts -> service.ts -> repository.ts -> validators.ts -> types.ts

This pattern keeps HTTP transport, business behavior, persistence, validation, and contracts separated. It also gives future developers a predictable place to add new behavior.

5. System Architecture

ProcureX is implemented as a monorepo with separate workspaces for the frontend client, backend server, and shared contracts. The architecture supports independent growth of user interface features, API modules, database models, and future intelligence services. 

procurement-platform/
|-- client/       React, Vite, routing, state, UI implementation
|-- server/       Express APIs, Prisma, modules, database access, tests
|-- shared/       Shared enums and contracts
|-- ml-service/   Reserved for future Python intelligence workloads
|-- docs/         Architecture, database, report, UI parity assets
|-- scripts/      Page generation and screenshot capture scripts
`-- package.json  Workspace scripts

5.1 Architectural Principles

  • The reference UI is not served directly by production.
  • The React client owns production user interaction and route behavior.
  • The Express server owns API routing, module registration, and backend orchestration.
  • The Prisma schema is the source of database model truth.
  • Organization-aware security is enforced through identity rules and RLS foundations.

8. Database And Security Implementation

The database layer uses Prisma as the application model and migration source. The schema models identity, organizations, verification, tenders, bids, evaluation, award recommendations, approvals, contracts, purchase orders, invoices, documents, audit events, compliance cases, risk signals, communication items, records, market intelligence, and integration records.

8.1 Core Data Rules

  • Accounts are only USER or ADMIN.
  • A normal user belongs to a company through OrganizationMember.
  • A company can act as buyer, supplier, or both through OrganizationCapability.
  • Tender ownership uses buyerOrgId.
  • Bid ownership uses both buyerOrgId and supplierOrgId.
  • Evaluation permissions use workflow assignments and RLS-aware access.

8.2 Row-Level Security

Row-level security protects procurement data by applying database access rules that respect current account and organization context. RLS SQL lives under server/prisma/sql/rls/, and the server database utilities live under server/src/db/. 

server/prisma/schema.prisma
server/prisma/sql/rls/001_app_context.sql
server/prisma/sql/rls/002_enable_rls.sql
server/src/db/context.ts
server/src/db/prisma.ts

8.3 Product Security Model

The most important security decision is that buyer and supplier are not account types. They are organization capabilities. This makes it possible for the same company to buy, supply, or do both while still keeping platform administrators separate from procurement operators.

9. Module-By-Module Workflow Implementation

ProcureX is organized by workflow modules. Each workflow has a visible user interface and a backend/data model boundary that supports future production behavior.

9.1 Procurement Lifecycle

Procurement covers opportunity discovery, tender creation, tender publication, tender details, tender documents, supplier tender views, and procurement guidance.

9.2 Bidding And Evaluation

Bidding supports supplier response preparation, bid versions, documents, responses, and receipts. Evaluation supports scoring workspaces, criteria, evaluator assignments, stages, and award recommendation handoff.

9.3 Awards And Contracts

Awards and contracts convert evaluation outcomes into recommendations, approvals, contract versions, purchase orders, invoices, and post-award tracking.

9.4 Communication And Records

Communication and records support collaboration and traceability. They preserve procurement messages, attachments, record entries, and audit evidence.

9.5 Administration And Compliance

Administration supports platform oversight. Admin accounts handle verification, compliance review, user management, audit review, analytics, and search. Admin is a platform compliance role and is separate from buyer or supplier operation.

12. Conclusion And Recommendations

The ProcureX implementation establishes a strong foundation for a production procurement platform. It combines a visual reference workflow, production React pages, modular Express APIs, Prisma database modeling, and organization-aware access rules. The design allows the product to preserve the experience of the original prototype while moving toward a maintainable and secure application architecture.

12.1 Conclusion

The implementation successfully models the major procurement lifecycle areas: public discovery, onboarding, verification, procurement planning, tender management, bidding, evaluation, awards, contracts, communication, records, and administration. The separation of account type from organization capability is especially important because it supports realistic procurement operations where companies may act as buyers, suppliers, or both.

12.2 Recommendations

  1. Continue converting prototype-derived screens into richer production React workflows where backend APIs are ready.
  2. Expand module tests as each domain moves from skeleton status to business behavior.
  3. Keep UI parity screenshots updated during design or layout changes.
  4. Strengthen RLS verification as more sensitive procurement records become active.
  5. Introduce the future intelligence service only after stable integration contracts are defined.
